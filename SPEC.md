## 1. 项目概述

### 1.1 项目目标

从零手动实现完整的 RAG（检索增强生成）链路，不依赖 LangChain、LlamaIndex 等现成框架，帮助理解 RAG 的核心原理。

### 1.2 核心价值

| 特性 | 说明 |
|------|------|
| **零框架依赖** | 所有组件手工实现，便于学习原理 |
| **中文优化** | 使用阿里云 DashScope（通义千问）中文模型 |
| **轻量级** | 核心代码约 2000 行，适合教学和快速原型 |
| **可扩展** | 模块化设计，便于替换底层组件（如 FAISS、Milvus） |

## 3. 功能模块详解

### 3.1 文档加载 (document_loader.py)

| 功能 | 说明 |
|------|------|
| 支持格式 | TXT, DOCX, XLSX, PPTX, PDF, HTML |
| 分块策略 | 固定字符数 + 重叠 + 句子边界切割 |
| 默认参数 | chunk_size=500, overlap=100 |
| 元数据 | 记录文件名、修改时间、大小 |

**分块算法特点**：
- 优先在句子边界（。！？\n）切割
- 保留上下文重叠，避免关键信息被切断
- 中文优化（使用 jieba 分词）

### 3.2 文本嵌入 (embedding.py)

| 功能 | 说明 |
|------|------|
| 模型 | text-embedding-v2 (1536 维) |
| API | DashScope TextEmbedding |
| 分批 | 每批最多 10 条（API 限制） |
| 错误处理 | 失败时抛出 RuntimeError |

### 3.3 向量存储 (vector_store.py)

| 功能 | 说明 |
|------|------|
| 存储方式 | 内存 (numpy array) |
| 相似度算法 | 余弦相似度 |
| 持久化 | vectors.npy + chunks.json + sources.json |
| BM25 | 自实现 BM25，支持增量添加 |
| 融合算法 | RRF (Reciprocal Rank Fusion, k=60) |

**RRF 融合公式**：
```
RRF_score(d) = Σ (1 / (k + rank_i(d)))
```
其中 k=60，rank_i(d) 是文档 d 在第 i 个检索器中的排名。

### 3.4 检索器 (retrieval.py)

| 功能 | 说明 |
|------|------|
| Query 改写 | 用 LLM 优化模糊/简短问题 |
| 混合检索 | 向量 + BM25，权重可配置 |
| Rerank | 用 LLM 对候选结果重新排序 |
| 默认 fetch_k | top_k × rerank_factor (用于 rerank 候选) |

### 3.5 RAG 流水线 (rag_pipeline.py)

| 功能 | 说明 |
|------|------|
| 知识库构建 | 加载文档 → 分块 → Embedding → 存储 |
| 知识库加载 | 从磁盘加载已有向量库 |
| 增量更新 | 检测文档变更，仅更新变更部分 |
| 变更检测 | 对比文件 mtime 和 size |
| 回答生成 | Prompt 构建 + LLM 调用 |

### 3.6 Web 界面 (app.py)

| 功能 | 说明 |
|------|------|
| 界面风格 | ChatGPT 风格，左侧文档列表 + 右侧对话 |
| 文档上传 | 支持多文件上传 |
| 知识库管理 | 查看文档列表、清空知识库 |
| 实时统计 | 显示文档数和文本块数 |

## 6. 后续改进方向

### 6.1 高优先级

#### 6.1.1 向量数据库优化

**目标**：支持大规模文档库（>10000 块）

**方案选择**：

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| FAISS | 性能快 100 倍，Facebook 开源 | 需额外安装 | ⭐⭐⭐⭐⭐ |
| Chroma | 功能全面，API 友好 | 性能一般 | ⭐⭐⭐⭐ |
| Milvus | 分布式支持，适合企业 | 部署复杂 | ⭐⭐⭐ |
| Qdrant | Rust 实现，高性能 | 需 Docker 部署 | ⭐⭐⭐⭐ |

**推荐路径**：
1. 当前：numpy 内存实现（已满足 <1000 块场景）
2. 中期：切换到 FAISS（改动小，性能提升大）
3. 长期：可选 Qdrant/Chroma（功能更全面）

#### 6.1.2 API 调用优化

**目标**：提高系统鲁棒性，减少 API 调用成本

**改进项**：
- 添加指数退避重试机制（最大 3 次重试）
- Embedding 结果缓存（基于文档哈希）
- LLM 调用超时控制（30s）
- 请求速率限制（RPM 控制）

### 6.2 中优先级

#### 6.2.1 检索评估指标

**目标**：量化评估 RAG 系统性能

**评估指标**：

| 指标 | 说明 | 测试方法 |
|------|------|----------|
| 召回率 (Recall) | 检索结果中包含正确答案的比例 | 人工标注数据集 |
| MRR (Mean Reciprocal Rank) | 首个正确答案排名的倒数均值 | 自动化测试 |
| NDCG | 考虑排名位置的评估指标 | 自动化测试 |
| 回答准确率 | LLM 生成答案的正确率 | 人工评估或 GPT-4 评估 |

**测试数据集**：
- 准备 50-100 个问答对
- 覆盖不同类型问题（事实型、解释型、推理型）
- 包含边界情况（无答案、多答案）

#### 6.2.2 缓存机制

**目标**：减少重复 API 调用

**缓存项**：

| 缓存项 | 缓存 Key | 缓存时间 | 说明 |
|--------|----------|----------|------|
| Embedding 结果 | 文档块哈希 | 文档未变更 | 基于文本 SHA256 |
| Query 改写结果 | 问题哈希 | 24 小时 | 避免重复 LLM 调用 |
| 检索结果 | Query + top_k 哈希 | 知识库未变更 | 加速多轮对话 |
